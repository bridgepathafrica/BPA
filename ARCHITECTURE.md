# Architecture — Bridgepath Africa

> System design reference for engineers and technical stakeholders.

---

## System overview

Bridgepath Africa is a full-stack TypeScript monorepo with a clear three-tier architecture:

```
┌─────────────────────────────────────────────────────────────────┐
│  Browser (React SPA)                                            │
│  ·  Vite 7 + React 19 + TailwindCSS 4                          │
│  ·  TanStack Query for server state + optimistic updates        │
│  ·  Wouter for client-side routing                              │
└───────────────────────────┬─────────────────────────────────────┘
                            │  HTTP / JSON  (proxied in dev)
                            │  /api/* routes
┌───────────────────────────▼─────────────────────────────────────┐
│  API Server (Express 5 / Node.js 24)                            │
│  ·  HMAC-signed session tokens (TOKEN_SECRET)                   │
│  ·  Role-based middleware (job_seeker | employer)               │
│  ·  Rate limiting per route group                               │
│  ·  Helmet.js security headers                                  │
│  ·  Pino structured JSON logging                                │
└───────────────────────────┬─────────────────────────────────────┘
                            │  Drizzle ORM
┌───────────────────────────▼─────────────────────────────────────┐
│  PostgreSQL                                                     │
│  ·  5 core tables (users, profiles, jobs,                       │
│     applications, contact_submissions)                          │
│  ·  Managed via Drizzle push / generate                         │
└─────────────────────────────────────────────────────────────────┘
```

```
│
├── artifacts/api-server/         @workspace/api-server

│   Bundles with esbuild to a single file (dist/index.js).

├── lib/db/                       @workspace/db
│   Drizzle ORM schema definitions and migration runner.
│   Shared by the API server; never imported by the frontend.
│
├── lib/api-spec/                 @workspace/api-spec
│   OpenAPI 3.1 specification (openapi.yaml).
│   Source of truth for request/response shapes.
│
├── lib/api-client-react/         @workspace/api-client-react
│   TanStack Query hooks auto-generated by Orval from openapi.yaml.

├── lib/api-zod/                  @workspace/api-zod
│   Zod schemas auto-generated by Orval from openapi.yaml.
│   Used by the API server for runtime request validation.
│
└── scripts/                      @workspace/scripts
    Database seed and maintenance utilities.
```

## Authentication flow

```
User submits email + password
         │
         ▼
POST /api/auth/login
  1. Look up user by email in `users` table
  2. Hash submitted password with PBKDF2 + PASSWORD_SALT
  3. Compare hash — 401 if mismatch
  4. Generate token: base64url( userId : issuedAt : HMAC-SHA256 )
  5. Return { token, user } — 200
         │
         ▼
Frontend stores token in localStorage["bridgepath_token"]
  · Client-side expiry check on every app load (90-day TTL)
  · Token sent as Authorization: Bearer <token> on every API call
         │
         ▼
Protected API routes
  · requireAuth middleware parses and verifies the HMAC
  · Attaches req.user = { id, email, role }
  · Returns 401 if token invalid or expired
```

### Demo accounts

Two hard-coded demo users (IDs 0 and -1) are resolved entirely in the frontend `auth.tsx` without any server call. Demo sessions never touch the database. Mock job data (IDs 101–112) is injected by the frontend for demo users only.

## Data model

```
users
  id          SERIAL PRIMARY KEY
  email       TEXT UNIQUE NOT NULL
  password    TEXT NOT NULL          -- PBKDF2 hash
  role        TEXT NOT NULL          -- 'job_seeker' | 'employer'
  created_at  TIMESTAMPTZ

profiles
  id          SERIAL PRIMARY KEY
  user_id     INTEGER → users.id
  name        TEXT
  bio         TEXT
  location    TEXT
  country     TEXT
  skills      TEXT[]
  avatar_url  TEXT
  cv_url      TEXT
  linkedin    TEXT
  website     TEXT
  ...

jobs
  id          SERIAL PRIMARY KEY
  employer_id INTEGER → users.id
  title       TEXT
  location    TEXT
  type        TEXT                   -- 'full_time' | 'contract' | ...
  salary_min  INTEGER
  salary_max  INTEGER
  currency    TEXT
  description TEXT
  status      TEXT                   -- 'active' | 'closed'
  created_at  TIMESTAMPTZ

applications
  id          SERIAL PRIMARY KEY
  job_id      INTEGER → jobs.id
  user_id     INTEGER → users.id
  cover_letter TEXT
  status      TEXT                   -- 'pending' | 'shortlisted' | 'rejected'
  created_at  TIMESTAMPTZ

contact_submissions
  id          SERIAL PRIMARY KEY
  name        TEXT
  email       TEXT
  subject     TEXT
  message     TEXT
  created_at  TIMESTAMPTZ
```

## API design principles

1. **Single spec source of truth** — all shapes live in `lib/api-spec/openapi.yaml`. Client hooks and Zod validators are generated from it; do not hand-write them.
2. **Role-scoped endpoints** — the `requireAuth` + role-check middleware gates routes; never rely on the client to enforce roles.
3. **Consistent error shape** — all error responses follow `{ error: string, details?: unknown }`.
4. **Rate limiting** — three tiers: strict (auth endpoints, 10 req/15 min), standard (mutations, 60 req/min), generous (reads, 300 req/min).
5. **No secrets in responses** — password hashes and internal IDs never leave the server in API responses.

### State management

- **Server state** — TanStack Query (via generated hooks in `@workspace/api-client-react`)
- **Auth state** — React context (`AuthProvider` in `lib/auth.tsx`)
- **UI state** — local `useState` / `useReducer` per component; no global UI store

## Generated code — do not edit manually

The following directories contain auto-generated code. Always regenerate after changing the OpenAPI spec:

```bash
pnpm --filter @workspace/api-client-react run generate
pnpm --filter @workspace/api-zod run generate
```

Generated directories:
- `lib/api-client-react/src/generated/`
- `lib/api-zod/src/generated/`
